AI Usage Report: Personal Task Tracker Development
Огноо: 2026 оны 5-р сарын 6
Хөгжүүлэгч: Лувсанжал
Төслийн нэр: Personal Task Tracker (Node.js + Express + SQLite)
1. Оршил (Introduction)
Энэхүү тайлан нь "Бие даалт 13"-ын хүрээнд AI (Claude Code) хэрэгслийг хөгжүүлэлтийн явцад хэрхэн ашигласан туршлагаа задлан шинжлэх зорилготой.
Хөгжүүлэлтийн гол зарчим нь "Verify, don't trust" (Шалга, бүү итгэ) байсан бөгөөд AI-ийг зөвхөн үржүүлэгч (multiplier) болгож ашиглахыг хичээсэн.
2. Ажил үүргийн хуваарилалт (Division of Labor)AI болон миний хооронд хийгдсэн ажлуудын зааг ялгаа нь суралцах үйл явцыг алдагдуулахгүй байхад чиглэгдсэн.
2.1. Төлөвлөлтийн шат (Part A - Plan)AI-ийн гүйцэтгэсэн үүрэг: Стекүүдийн харьцуулалт (Node.js, Python, Go) болон Mermaid диаграммын анхны ноорог (draft) гаргах, ADR-ийн форматыг бэлдэх. 
Миний гүйцэтгэсэн үүрэг: Технологийн эцсийн сонголт (Node.js), CLAUDE.md дээрх "No-go zones" (AI-аар хийлгэж болохгүй ажлууд) тодорхойлох, төслийн хамрах хүрээг (scope) хязгаарлах. 
2.2. Хөгжүүлэлтийн шат (Part B - Build)AI-ийн гүйцэтгэсэн үүрэг: Express boilerplate кодын загвар, Zod validation-ий схем, тест хийх "edge case"-уудын саналууд.  
Миний гүйцэтгэсэн үүрэг: Бизнес логикийн дүрмүүдийг (жишээ нь: "Done" болсон ажлыг "Todo" руу буцааж болохгүй) шийдэх, Өгөгдлийн сангийн нормалчлал (many-to-many), санал болгосон тест бүрийг уншиж, шаардлагагүйг нь хасах.  
3. AI-ийн алдаа ба "Hallucination" (Оношлогоо ба Засвар)AI-тай ажиллах явцад гарсан алдаануудыг олж зассан нь миний суралцах үйл явцын хамгийн чухал хэсэг байлаа.
Жишээ 1: Байхгүй npm багц санал болгох
AI express-rate-limiter гэдэг багцыг санал болгосон ч бодит нэр нь express-rate-limit байсан.  
Шалгасан арга: Терминал дээр npm install ажиллуулахад 404 алдаа гарсан тул npmjs.com дээрээс зөв нэрийг хайж олсон.  
Жишээ 2: Express 4-т байхгүй метод ашиглах
AI app.useAsync() гэдэг метод ашиглахыг санал болгосон ч энэ нь Express 4 дээр байхгүй байв.  
Зассан арга: Би өөрөө asyncHandler wrapper функц бичиж, алдааг next(err)-ээр дамжуулдаг болгож зассан.  
4. Аюулгүй байдал ба "Verify, don't trust"AI-ийн санал болгосон хамгийн эрсдэлтэй код нь SQL хайлт хийх хэсэг байв.
AI-ийн санал: Template literal ашигласан шууд SQL query.  JavaScriptdb.prepare(`SELECT * FROM tasks WHERE title LIKE '%${query}%'`).all(); // АЮУЛТАЙ
Миний засвар: SQL injection-оос хамгаалахын тулд Prepared Statements болон ? placeholder ашигласан.  
JavaScriptdb.prepare('SELECT * FROM tasks WHERE title LIKE ?').all(`%${query}%`); // ЗӨВ
Илрүүлсэн арга: /security болон /review slash command-уудыг ашиглан шалгаж, эрсдэлийг баталгаажуулсан.  
5. Үр ашиг ба Бэрхшээлүүд5.1. Хурд нэмэгдсэн хэсэг (Productivity Benefit)Boilerplate тохиргоо, тестийн "edge case"-уудыг олох, OpenAPI ба Mermaid диаграммын синтакс бичихэд ойролцоогоор 6-8 цаг хэмнэсэн.  
5.2. Саад бэрхшээл (Slowdowns)Бизнес логикийг AI-д тайлбарлах, урт чат сессүүдийн үед AI өмнөх контекстээ (жишээ нь: ESM ашиглах) мартах зэрэг асуудлууд нь хөгжүүлэлтийг удаашруулж байсан.  
6. Ур чадвараа хадгалах стратеги (Skill Atrophy Management)AI-д хэт найдахаас сэргийлж дараах арга хэмжээнүүдийг авсан:AI-гүй өдрүүд: Хөгжүүлэлтийн зарим өдөр AI-ийг бүрэн хааж, зөвхөн албан ёсны баримт бичиг (docs) уншиж ажилласан. 
Line-by-line Review: AI-ийн үүсгэсэн код бүрийг мөр мөрөөр нь тайлбарлаж чадахаар болтол нь судалсан.  
Reflective Practice: learning-notes.md файлд шинээр сурсан зүйлсээ (RegEx, SQLite API г.м) тэмдэглэж үлдээсэн.
Дүгнэлт:
AI бол үржүүлэгч хэрэгсэл болохоос биш мэдлэгийг орлогч биш юм. 
Энэ төсөл нь надад "Mental Model"-оо хадгалахын зэрэгцээ орчин үеийн AI хэрэгслүүдийг аюулгүй, үр ашигтай ашиглаж сурахад томоохон хичээл боллоо. 
AI-тай ажиллахдаа "Verify, don't trust" зарчмыг баримтлах нь хамгийн чухал бөгөөд энэ нь зөвхөн кодын аюулгүй байдлыг хангаад зогсохгүй, мөн миний хөгжүүлэлтийн ур чадварыг хадгалахад тусалсан.
