## 1. Хэрэв шалгалт өнөөдөр болбол би энэ кодыг өөрөө бичиж чадах уу?

**Хариулт: Хэсэгчлэн (~70%)**

### ✅ Чадах хэсэг
- **Express boilerplate** — `app.js`, route, controller pattern сайн ойлгосон. Шинэ routes нэмэх боломжтой.
- **better-sqlite3-ийн API** — `prepare`, `run`, `all`, `get` ялгааг тайлбарлаж чадна.
- **Layer architecture** — Routes → Controller → Service → Repository чиглэлийн логикийг ойлгож, шинэ feature нэмэхэд аль layer-д юу нэмэхээ мэднэ.
- **Conventional Commits ба git workflow** — өдөр тутамд commit format-ыг өөрөө дагаж чадна.
- **OWASP Top 10-ийн үндсэн ангилал** — `/security` slash command-аас сурсан зүйлсээ хадгалсан.

### ⚠️ Эргэлзээтэй хэсэг
- **Zod schema-ийн нарийн** — `refine`, `transform`, `union` ашиглах гэх мэт ахицлаг feature-уудыг өөрөө шинээр бичих эргэлзээтэй. Жишээ хайхгүйгээр санах боломжгүй.
- **Mermaid syntax** — энгийн flowchart бичиж чадах ч `sequenceDiagram` дээрх `alt`, `loop`-ийг санах эргэлзээтэй.
- **Express error middleware**-ийг яг тэр хэлбэрээр санах — 4-р параметр `err` гэдгийг санах боловч, signature шалгахгүй бол typo гарч магадгүй.

### ❌ Чадахгүй (АІ-аар хэт удаан түшсэн хэсгүүд)
- **Regex pattern** — input validation-д ашигласан regex-уудыг ихэвчлэн AI бичсэн. Шинээр regex бичих чадвар муудсан.
- **OpenAPI yaml-ийн нарийвчилсан schema definition** — `$ref`, `oneOf` ашиглах нь өөрсдөө хайх хэрэгтэй болно.

## 2. Дахин хийнэ гэвэл юуг өөрөөр хийх вэ?

### Гол өөрчлөлт
1. **CLAUDE.md-ийг өдөр бүр шинэчлэх биш, эхэндээ их хэмжээгээр хийхгүй.** Энэ удаа CLAUDE.md эхэндээ 350+ мөртэй болж AI зөвлөмжийг алгасах болсон. Бага оптимизаци хийсэн дүрмүүдийг л оруулах байсан.

2. **AI-гүй өдрүүдийг **эхний** долоо хоногт хуваарилах** — Build-ийн дунд хэсэгт AI-ийг эрчимтэй ашиглахаас өмнө үндэс муу талыг бэхжүүлэх байсан. Энэ удаа AI-гүй өдрүүдийг 4, 7, 11 дэх өдрүүдэд оруулсан — хэт хожуу.

3. **Test-ыг feature-аас өмнө бичих** (TDD). Энэ удаа feature implement → test гэж явсан учраас зарим test "passes-ийн төлөө" бичигдсэн. Хэрэв test-ыг урьдчилан бичсэн бол AI-ийн санал болгосон код илүү тодорхой шаардлагатай нийцэх байсан.

4. **AI session log-ыг чат явагдах үед бичих**, дараа нь summary хийх биш. Сүүлд эргэн хайхад чатын дунд алдсан зарим чухал шийдвэр алга болсон.

5. **Жижиг commit илүү олон.** 15 commit л хийх биш, 25-30 commit хийх. Жишээ нь "feat(tasks): add create endpoint" + "test(tasks): add create endpoint tests" 2 commit болгож хувааж болох байсан.

6. **Үг тоо бус, чанар гэж бодох.** AI Usage Report-ийн 1500 үгийн threshold-ыг "хэт олон үг бичиж дүүргэх" гэж бодсон тал бий. Дахин хийвэл нэг hallucination жишээг 200 үгээр гүн бичих байсан.

### Жижиг өөрчлөлт
- TypeScript-ийг шууд эхнээс ашиглах (zod schema-аас type infer хийх).
- `.editorconfig` файл нэмэх (consistency).
- `husky` + `lint-staged` pre-commit hook-той болгох.
- GitHub Actions CI ажиллуулж тестийг pull request бүрд шалгах.

## 3. Энэ туршлагаас юу сурсан бэ?

### Техникийн талаас
1. **AI бол үржүүлэгч, бэлтгэл биш.** Боловсорсон ур чадвартай бол AI-аар хурдан ажиллана. Бэлтгэлгүйгээр AI байсан ч "ажиллах" код гарна, гэвч ойлгохгүй.
2. **CLAUDE.md ба slash commands бол project-ын файл, хувийн биш.** Эдгээрийг repo-д commit хийсэн нь өөр оюутан / ирээдүйн би өөрөө ашиглахад тус болно.
3. **Layered architecture бол overkill биш.** Эхэндээ "жижиг проектод 4 layer хэт олон" гэж бодсон ч test-ыг service-аас mock repository-аар тестлэх боломж нь утгатай байсан.
4. **OpenAPI spec-ийг гар аргаар бичих нь сайн дасгал.** Үүнээс хойш REST API дизайн хийхэд route-ы талаар тодорхой бодох болсон.

### Process талаас
1. **"Verify, don't trust"** — лекц 13-аас сурсан энэ зарчмыг практикт хэрэглэхэд яагаад чухал болохыг ойлгов. Express `useAsync` hallucination нь нэг өдрийг идсэн. AI хариунд скептик хандаж асуудал тэмдэглэх дадал.
2. **Conventional Commits** нь зөвхөн стандарт биш — git history-ийг товчоор унших боломжтой болгодог. Бие даалт-ын review хийхэд "юу өөрчлөгдсөн" гэдгийг хурдан харж болно.
3. **AI session log хадгалах нь reflective practice.** Дахин эргэн харахад "AI ийм асуудалтай байсан, бид яаж шийдвэрлэсэн" гэдгийг ярих чадвартай.

### Хувийн талаас
1. **Хариулт хүлээх биш — асуулт сайжруулах.** AI-ийн хариулт нь асуулт чанартай уялдсан. "Энэ зөв үү?" гэхээс "Энэ ямар edge case дээр fail болж болох вэ?" гэж асуухад илүү гүн хариу авдаг.
2. **AI-аар хийсэн кодыг өөрөө тайлбарлах сорилт** нь скилл атрофи-аас хамгаалахын хамгийн сайн арга. Жишээ нь өөрийн кодыг найз руу explain хийх. Тайлбарлахад дутуу мэдрэмж бий бол — мэдэхгүй гэсэн үг.
3. **Документ бичих ур чадвар** — энэ бие даалт нь миний хувьд code бичихээс илүү документ бичсэн. ARCHITECTURE.md, ADR, AI Usage Report — эдгээр нь ирээдүйн ажилд хамгийн чухал ур чадвар.
